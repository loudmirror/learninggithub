# Epic: LearningGitHub MVP v0.1

> **Epic ID:** EPIC-MVP-001
> **Status:** Draft
> **Created:** 2025-11-20
> **Target Completion:** 7-8 weeks
> **Priority:** P0 - Critical Path

---

## Epic Goal

构建 LearningGitHub 的最小可行产品（MVP），使中文用户能够通过简单粘贴 GitHub 仓库链接，自动获得母语化的项目学习教程，包括项目概览、模块化学习路径、手把手运行步骤和基础问答能力，验证产品价值假设。

## Epic Description

### 产品愿景

LearningGitHub 是一个 AI 驱动的 GitHub 项目学习助手，致力于打破语言和技术门槛，让任何人都能轻松理解和学习开源项目。

### MVP 范围

**核心价值主张：**
- 🌐 **语言桥梁**：将英文文档转换为通俗易懂的中文说明
- 🗺️ **结构导航**：指出关键文件和模块，建立整体认知
- 📚 **路径引导**：提供从环境准备到项目运行的模块化学习路径
- 💬 **交互学习**：基于项目上下文的智能问答

**技术架构：**
- **后端**：Python + FastAPI
- **前端**：React / Next.js
- **向量数据库**：ChromaDB（嵌入式）
- **AI 服务**：OpenAI API（LLM + Embeddings）
- **数据源**：GitHub REST API + Git

**功能边界：**
- ✅ 支持公开仓库解析和教程生成
- ✅ 中文为主要输出语言
- ✅ 项目概览、结构说明、运行教程、学习路径、基础问答
- ❌ 不支持私有仓库（MVP 阶段）
- ❌ 不包含游戏化、社区、多用户账号系统

### 目标用户

1. **编程初学者**：具备基础命令行和 Git 操作，想照着开源项目练手，但英语一般
2. **非技术人员**：产品/运营/设计师，希望了解项目功能和使用方式
3. **进阶开发者**：想快速评估新技术栈项目的结构和运行方式

### 成功标准（MVP 验证指标）

- ✅ 至少 10-20 名真实用户使用产品完成 1 个以上项目的学习流程
- ✅ ≥60% 的用户认为比直接看 GitHub 更容易理解项目
- ✅ ≥60% 的用户认为更容易按步骤跑起来项目
- ✅ 每个新用户至少发起 2 次以上问答交互

## Stories / Iterations

本 Epic 分为三个主要迭代，每个迭代对应一个或多个用户故事：

### 迭代 1：基础架构 + Mock 数据端到端（2-3周）

**目标**：打通端到端用户体验，使用 Mock 数据验证完整流程

**Story 1.1：搭建后端基础架构**
- 创建 FastAPI 项目骨架
- 实现 `GET /api/tutorial` Mock 接口
- 配置统一的错误处理和日志系统
- 验收：Mock 接口返回符合规范的教程数据结构

**Story 1.2：创建前端基础应用**
- 创建 React/Next.js 项目
- 实现 Home 页面（URL 输入、语言选择）
- 实现 Tutorial 页面（概览、结构、学习路径、步骤展示）
- 验收：用户可以输入 URL 并看到完整的教程页面（Mock 数据）

**Story 1.3：实现学习路径 UI 交互**
- 模块列表展示与高亮
- 步骤完成标记
- 当前模块步骤过滤
- 验收：用户可以按模块导航和标记步骤完成

---

### 迭代 2：真实服务集成（3-4周）

**目标**：接入 GitHub、ChromaDB 和 LLM，生成真实的教程内容

**Story 2.1：GitHub 仓库解析服务**
- 实现 repoUrl 解析和校验
- 集成 GitHub REST API（获取元数据、README）
- 实现 git clone 和本地仓库缓存
- 构建目录树和识别关键文件
- 验收：能够成功解析中小型公开仓库并提取关键信息

**Story 2.2：内容处理与向量化**
- 实现内容分段策略
- 集成 ChromaDB 和 Embedding 模型
- 为代码/文档生成向量并建立索引
- 验收：仓库内容成功索引到 ChromaDB

**Story 2.3：RAG 教程生成服务**
- 实现基于 RAG 的项目概览生成
- 实现运行前准备生成
- 实现项目结构与文件讲解生成
- 实现运行步骤生成
- 验收：能为真实仓库生成合理的中文教程

**Story 2.4：学习路径生成**
- 基于步骤和结构生成模块列表
- 定义模块顺序和关联步骤
- 前端切换到真实数据模式
- 验收：生成的学习路径清晰且符合学习逻辑

**Story 2.5：缓存与错误处理优化**
- 实现仓库解析结果缓存
- 完善错误码体系（REPO_NOT_FOUND、REPO_TOO_LARGE、RATE_LIMITED）
- 添加大小和文件数限制
- 验收：相同仓库二次访问快速返回，各类错误有清晰提示

---

### 迭代 3：问答功能 + 体验优化（2-3周）

**目标**：实现上下文问答能力，优化整体用户体验

**Story 3.1：问答服务实现**
- 实现 `POST /api/tutorial/qa` 接口
- 基于 ChromaDB 检索相关片段
- 构造提示词并调用 LLM 生成回答
- 验收：用户提问后能获得上下文相关的准确回答

**Story 3.2：问答 UI 集成**
- 实现 QnAPanel 组件
- 展示聊天历史和问答气泡
- 显示 Loading 和错误状态
- 验收：问答交互流畅，体验良好

**Story 3.3：学习路径体验优化**
- 优化模块展开/折叠策略
- 调整视觉层级和留白
- 优化移动端响应式布局
- 验收：信息密度适中，用户不会迷失位置

**Story 3.4：小范围用户测试**
- 邀请 5-10 名目标用户试用
- 收集关于学习路径、运行成功率、问答质量的反馈
- 根据反馈进行必要调整
- 验收：完成用户测试报告和优化清单

## 风险与缓解措施

### 🔴 高风险

**R1：生成教程质量不稳定**
- **影响**：用户按教程操作失败，产品价值未验证
- **缓解**：
  - 使用结构化提示词模板固定输出格式
  - 选择 3-5 个典型仓库作为测试基准
  - 基于用户反馈快速迭代

**R2：GitHub API 限流导致失败率高**
- **影响**：用户体验差，无法完成解析
- **缓解**：
  - 强制使用 GitHub Token 认证（5000次/小时）
  - 实现仓库解析结果缓存
  - 提供清晰的限流错误提示

### 🟡 中风险

**R3：向量检索相关性不够准确**
- **影响**：生成的教程内容不够精准
- **缓解**：
  - 优化分段策略（按文件/函数/类分段）
  - 调整检索参数（top_k、相似度阈值）
  - 增加元数据过滤

**R4：LLM 调用成本超预算**
- **影响**：MVP 阶段成本过高
- **缓解**：
  - 实施多级缓存（内存、Redis、数据库）
  - 限制单用户请求频率
  - 优化提示词长度

### 🟢 低风险

**R5：大仓库处理时间过长**
- **影响**：用户等待时间过长
- **缓解**：
  - 设置仓库大小上限（如 50MB）
  - 提供预计处理时间提示
  - 实现进度反馈机制

## Epic 完成定义 (Definition of Done)

- [ ] 所有迭代（1-3）的故事全部完成并验收
- [ ] 至少支持 3 个不同技术栈的仓库（Node.js, Python, 其他）
- [ ] 端到端用户流程测试通过（输入 URL → 查看教程 → 提问 → 获得回答）
- [ ] 所有核心 API 接口有基础的错误处理和日志
- [ ] 前端在桌面和移动端都能正常使用
- [ ] 完成 5-10 名真实用户的测试并收集反馈
- [ ] 性能指标达标：
  - 中小型仓库解析时间 < 90 秒
  - 问答响应时间 < 10 秒
  - 缓存命中后返回时间 < 5 秒
- [ ] 代码提交到 Git 仓库并有基础的 README
- [ ] 部署到可公开访问的测试环境
