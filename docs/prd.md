## 1. 背景与目标

### 1.1 项目背景

- GitHub 上有大量优质的开源项目，但对「不会写代码或英文一般的用户」来说，学习门槛非常高。
- 当前主流学习方式：
  - 阅读英文 README / 文档；
  - 直接读代码、在本地尝试运行；
  - 在搜索引擎或社区中查找教程。
- 这些方式普遍存在：
  - 语言障碍（英文文档阅读困难，尤其是技术英文）；
  - 结构复杂（不知道先看哪个文件、先做哪一步）；
  - 学习路径不清晰（缺少「从零到跑起来」的引导）。

### 1.2 问题陈述

对于普通用户，尤其是中文用户与编程初学者而言：

- 很难快速判断一个 GitHub 项目「是干什么的」「适不适合自己」；
- 不知道如何从一堆文件中找出关键文件和入口；
- 就算照着 README 操作，也常常遇到错误，缺乏清晰的排查指引；
- 很难围绕「一个具体项目」进行系统化学习，只能零散搜索。

### 1.3 产品愿景

> 让英文不好的新手小白都能看懂并动手实践任意 GitHub 项目。

LearningGitHub 致力于成为一个「GitHub 项目的中文学习助手」，通过自动解析仓库、生成手把手教程、提供问答支持，降低用户理解与实践开源项目的门槛。

### 1.4 目标与成功标准（MVP 阶段）

**产品目标（MVP）**

- 实现「输入 GitHub 仓库 URL → 输出中文项目概览 + 从零运行教程 + 文件/模块讲解 + 模块化学习路径」的完整闭环。
- 支持用户围绕「项目概览与运行步骤」提问并获得相关回答。
- 验证用户是否愿意以这种方式来学习 GitHub 项目。

**成功标准（MVP 验证性指标）**

- 至少 10–20 名真实用户使用产品完成 1 个以上项目的「从克隆到运行」学习流程。
- ≥ 60% 的试用用户在使用后认为：
  - 能比直接看 GitHub 页面「更快理解项目」；
  - 「更容易按步骤跑起来」。
- 每个新用户至少发起 2 次以上问答交互（证明他们在以此方式探索项目）。

## 2. 产品概述

### 2.1 一句话总结

> LearningGitHub 是一个帮你看懂任意 GitHub 仓库的中文学习助手，只要粘贴链接，就能获得用你母语写成的、手把手的项目学习教程。

### 2.2 核心价值主张

- **语言桥梁**：将英文 README/文档转换为通俗易懂的中文说明。
- **结构导航**：指出应先看哪些文件、了解哪些模块，建立整体认知。
- **路径引导**：提供从环境准备到项目成功运行的步骤式教程，并给出模块化学习路径。
- **交互学习**：用户可就项目提出问题，获得上下文相关的解释。

### 2.3 目标用户与 Personas

1. **编程初学者 A（大学生/自学者）**
   - 具备基础命令行和 Git 操作；
   - 想「照着开源项目练手」；
   - 英语一般，看 README 吃力。

2. **技术好奇的非工程人员 B（产品/运营/设计）**
   - 希望大致理解某个项目的功能与使用方式；
   - 不会深入阅读代码，只想知道「能不能用」「怎么用」。

3. **进阶开发者 C（对新技术栈感兴趣）**
   - 能读代码，但时间有限；
   - 希望快速评估一个项目的结构、运行方式和技术选型。

### 2.4 典型使用场景

- 场景 1：学生拿到课程作业，需要使用某个 GitHub 项目，但完全看不懂英文 README。
- 场景 2：产品经理想了解某个热门开源项目的功能范围和应用场景，以便评估集成价值。
- 场景 3：初学者想跟着一个例子项目动手练习，却不知道该按什么顺序执行命令。

## 3. 范围界定（MVP）

### 3.1 In Scope（MVP）

- 输入：单个公开 GitHub 仓库 URL。
- 输出：
  - 简洁的中文项目概览（用途、主要功能、技术栈）。
  - 从「克隆仓库」到「本地运行」的手把手教程。
  - 文件与模块讲解（基础版）：对核心目录和关键文件给出简要中文说明。
  - 模块化学习路径（基础版）：将项目拆成若干学习模块，给出推荐学习顺序。
- 基础问答：
  - 围绕项目概览、运行步骤和核心文件结构的问答。
- 支持平台：
  - Web 端应用（桌面浏览器优先，自适应移动端即可）。

### 3.2 Out of Scope（暂不在 MVP 范围内）

- 私有仓库分析（需要认证的情形先不支持）。
- 游戏化功能（等级、勋章、学习时长排行榜等）。
- 社区功能（公开讨论区、学习笔记分享）。
- 自托管/本地部署模式。
- 深度代码级别的调试指导与性能优化建议。

### 3.3 关键假设与约束

- 只处理公开仓库，使用 GitHub 官方 API 及/或 `git clone`。
- 优先支持常见栈（如 Node.js、Python）；其他语言以有限支持为主。
- 使用云端托管的大模型与向量服务，成本可控的前提下优先选择效果较好的方案。

## 4. 功能需求（Functional Requirements）

> 编号形式：FRx

### FR1 仓库 URL 解析与校验

- **FR1.1**：用户可以在首页输入 GitHub 仓库 URL。
- **FR1.2**：系统校验 URL 是否有效（格式正确、域名为 github.com）。
- **FR1.3**：若仓库不存在或无法访问，系统给出明确错误提示（如「仓库不存在」或「项目为私有」）。
- **FR1.4**：支持含分支、子目录的 URL，但 MVP 可先简化为主分支解析。

**验收要点**

- 非 GitHub URL 或格式错误时，前端即时提示。
- 对无效仓库，系统不会进入下一步生成流程。

---

### FR2 仓库信息拉取与内容抽取

- **FR2.1**：系统可通过 GitHub API 获取仓库基本信息：
  - 名称、描述、star 数、主要语言、创建时间等。
- **FR2.2**：系统可读取仓库 README（优先中文/英文版本）。
- **FR2.3**：系统可构建主要目录结构树，识别关键目录（如 `src`、`app`、`config` 等）。
- **FR2.4**：对于仓库体积在允许范围内，系统可拉取并解析关键代码/配置文件（如 `package.json`、`requirements.txt`、`Dockerfile` 等）。

**验收要点**

- 对于常见开源项目，能在 1 分钟内完成信息拉取与解析（不含生成时间）。
- 若仓库过大或文件类型不支持，系统提示并限制解析范围，而非失败崩溃。

---

### FR3 中文项目概览生成

- **FR3.1**：系统基于仓库信息与 README，生成一段中文项目概览，包含：
  - 这个项目是做什么的；
  - 主要功能点；
  - 面向的用户或使用场景；
  - 使用的主要技术栈。
- **FR3.2**：概览语言风格应尽量避免过多技术术语，对必需术语进行简单解释。
- **FR3.3**：概览长度控制在约 3–8 段/要点之间，方便快速阅读。

**验收要点**

- 对于典型项目，概览能让非工程人员在 1–2 分钟内知道项目大致用途。
- 用户评价概览「比直接读 README 容易理解」。

---

### FR4 手把手运行教程生成

- **FR4.1**：系统基于配置文件和文档，生成「从 0 到项目跑起来」的步骤式教程：
  - 环境准备（语言版本、依赖、工具等）；
  - 仓库克隆；
  - 依赖安装；
  - 环境变量配置（如需）；
  - 启动命令；
  - 验证应用是否正常运行的方法。
- **FR4.2**：每个步骤包含：
  - 要执行的命令（以代码块形式展示）；
  - 对该步骤目的的简短说明；
  - 预期结果（如终端输出片段、网页访问路径等）。
- **FR4.3**：对于常见错误（如依赖安装失败、端口占用），提供一两条常见排查建议。
- **FR4.4**：教程默认以简体中文呈现。

**验收要点**

- 找 3 个真实用户，选 3 个常见项目；在不看 README 的情况下，仅靠教程能成功跑起来 ≥ 80%。
- 教程能在页面上以清晰的步骤列表展示，并可滚动查看。

---

### FR5 教程展示与导航界面

- **FR5.1**：系统提供一个教程展示页面，包含至少三个区域：
  - 项目基本信息与概览；
  - 运行教程步骤列表；
  - 问答区域（见 FR6）。
- **FR5.2**：运行教程支持「折叠/展开」步骤，便于聚焦当前进行的部分。
- **FR5.3**：当前正在进行的步骤有明显视觉标记（例如高亮或编号强调），用户可以手动点击标记「完成」。

**验收要点**

- 用户在使用时不会迷失当前进行到哪一步。
- 对于 5 个以上步骤的教程，交互仍清晰、可操作。

---

### FR6 基础问答功能（围绕项目与教程）

- **FR6.1**：用户可以在教程页面通过一个输入框提出问题，问题内容可以为自然语言中文。
- **FR6.2**：系统根据仓库内容和已生成的教程，为用户返回一段回答，回答应尽量具体且贴近上下文。
- **FR6.3**：问题类型示例：
  - 「这个项目用的是什么框架？」
  - 「我应该先看哪个文件？」
  - 「这个运行命令是做什么的？」
- **FR6.4**：回答中引用的文件名或命令应尽量可点击或容易复制。

**验收要点**

- 针对简单问题（框架、语言、入口文件），答案准确率高。
- 用户反馈问答能「补充教程中不够清楚的地方」。

---

### FR7 项目记录与历史访问（基础）

- **FR7.1**：系统记录用户最近访问的若干个仓库（无需强制登录，基于浏览器本地存储即可）。
- **FR7.2**：用户在首页可看到最近访问项目列表，点击可快速回到对应教程页面。

**验收要点**

- 用户刷新页面或短时间内重新进入网站，仍能方便地回到之前的学习项目。

---

### FR8 多语言支持（MVP 级别）

- **FR8.1**：系统核心输出为简体中文。
- **FR8.2**：为未来扩展预留语言选择接口（如英文）；MVP 不要求完整实现，但在架构设计中考虑语言维度。

**验收要点**

- 代码与接口设计中，文本生成部分以「语言参数」为可选项实现，而非硬编码中文。

---

### FR9 文件与模块讲解（基础版）

- **FR9.1**：系统基于仓库的目录结构和文件内容，为核心目录和关键文件生成简要中文说明（用于什么、在项目中的角色）。
- **FR9.2**：在教程页面提供「项目结构/模块概览」区域，以列表或树状形式展示这些目录/文件说明。
- **FR9.3**：用户能够通过点击目录/文件项，快速定位到相关步骤或代码片段（例如滚动到对应教程模块）。

**验收要点**

- 至少对核心目录（如 `src`、`app`、`config` 等）和入口文件生成描述。
- 非技术用户阅读后能大致理解「项目分成哪些部分，每部分负责什么」。

---

### FR10 模块化学习路径（基础版）

- **FR10.1**：系统将整套教程按主题/目标拆分成若干学习模块（例如「环境准备」「跑通项目」「了解核心功能」）。
- **FR10.2**：在教程页面展示模块列表及推荐顺序，并为每个模块提供简要说明和包含的关键步骤数量。
- **FR10.3**：用户可以标记模块为「已完成」，界面能展示整体学习进度（例如已完成模块数/总模块数）。
- **FR10.4**：模块划分与运行步骤相互关联，用户从模块跳转时能快速定位到对应的步骤区域。

**验收要点**

- 对于典型项目，模块数量适中（例如 3–7 个），名称清晰易懂。
- 至少 80% 的测试用户能理解模块顺序并按模块完成整个学习流程。

## 5. 非功能需求（Non-Functional Requirements）

> 编号形式：NFRx

### NFR1 性能

- **NFR1.1**：中小型仓库（例如 < 50 MB）的「解析 + 生成」整体时间在 30–90 秒内。
- **NFR1.2**：同一仓库再次访问时，如已完成解析与索引，应在 5 秒内返回结果（可用缓存）。

### NFR2 可用性与稳定性

- **NFR2.1**：出现错误时，界面要有清晰的错误文案，而不是空白页面或系统异常信息。
- **NFR2.2**：若 GitHub API 限流或网络异常，应提示用户稍后重试，而非无限等待。

### NFR3 可用性（Usability）

- **NFR3.1**：UI 文案尽量避免专业术语，对必要术语给出简短解释。
- **NFR3.2**：教程步骤格式统一，换行、代码块等排版清晰。

### NFR4 安全与隐私

- **NFR4.1**：不存储用户输入的敏感信息（如个人 token），MVP 阶段尽量仅使用公开 API。
- **NFR4.2**：如未来需要用户 GitHub Token，则需明确告知用途并妥善加密存储。

### NFR5 可维护性

- **NFR5.1**：核心功能应模块化（仓库解析、内容生成、问答），便于未来替换大模型或向量库。
- **NFR5.2**：关键流程需有基础日志记录，方便排查问题。

## 6. 用户体验与信息架构

### 6.1 核心页面与信息架构

1. **首页**
   - 主要元素：
     - GitHub URL 输入框
     - 语言选择（默认中文）
     - 最近访问项目列表
   - 操作：
     - 粘贴 URL → 点击「开始学习」。

2. **教程页面**
   - 顶部：项目名称、星数、语言、简要概览。
   - 左侧/上方：学习内容区域：
     - 项目概览（用途、主要功能、适用人群、技术栈）。
     - 项目结构与文件讲解（核心目录/关键文件的简要说明）。
     - 模块化学习路径视图（模块列表、顺序与状态）。
     - 分步运行教程（步骤列表）。
   - 右侧/下方：问答区域（问题输入 + 会话历史）。
   - 附加：返回首页或切换项目的导航。

3. **错误与状态页面/组件**
   - 加载中状态（解析与生成时的进度/提示）。
   - 常见错误展示（无效 URL、仓库过大、GitHub 限流等）。

### 6.2 交互与文案原则

- 尽量使用「你可以…」「下一步做…」的教学式语气。
- 在关键步骤提醒用户可能的坑（例如 Node 版本、端口占用）。
- 对于长时间生成过程，提供进度提示或「大致预计时间」。

## 7. 数据与事件（简要）

### 7.1 核心数据实体（逻辑概念）

- **RepositoryProfile**
  - repo_url
  - name / description / language / stars
  - 解析状态 / 创建时间 / 最近访问时间

- **Tutorial**
  - 关联 repo
  - 项目概览文本
  - 运行步骤（有序列表）
  - 生成时间 / 版本

- **LearningModule（学习模块，基础版）**
  - 关联 repo 或 tutorial
  - 模块标识（id）
  - 标题（例如「环境准备」「跑通项目」）
  - 简要描述
  - 显示顺序（order）
  - 关联的关键步骤或文件列表（可选）

- **Session / LearningSession（可选）**
  - 用户浏览器标识
  - 访问过的 repo 列表
  - 问答交互历史（可配置存储时长）

### 7.2 核心埋点事件（建议）

- `view_home`
- `submit_repo_url`（含 URL、是否成功解析）
- `view_tutorial`（含 repo 标识）
- `complete_step`（步骤编号）
- `view_learning_path`（查看模块化学习路径）
- `complete_module`（模块标识）
- `ask_question` / `receive_answer`
- `error_occurred`（错误类型）

## 8. 里程碑与发布计划（MVP）

### 8.1 阶段划分

1. **阶段 1：基础解析与静态教程**
   - 完成 URL 输入和仓库基础信息获取；
   - 基于 README 生成中文项目概览；
   - 前端展示项目概览页面。

2. **阶段 2：运行教程生成**
   - 增加仓库内容解析（关键文件）；
   - 引入向量/检索机制，生成运行步骤；
   - 生成基础的文件与模块讲解（核心目录、关键文件的职责说明）；
   - 设计并实现模块化学习路径结构（模块列表及其与步骤/文件的关联）；
   - 前端展示分步教程和项目结构/模块概览。

3. **阶段 3：问答与体验优化**
   - 实现基于检索的问答接口；
   - 在教程页面完善学习路径展示与交互（模块高亮、完成标记、模块级进度指示）；
   - 优化 UI（加载状态、错误提示、步骤标记）；
   - 收集试用用户反馈，调优提示词与教程结构。

### 8.2 对外发布策略

- MVP 阶段以「可控小范围内测」为主（同学、朋友、同行）。
- 根据使用情况和反馈迭代后，再考虑更大范围公开访问。

## 9. 主要风险与应对

- **风险 1：生成教程质量不稳定**
  - 应对：
    - 通过提示词与结构模板固定输出结构；
    - 选择典型仓库做手工对比和调优；
    - 基于用户反馈迭代。

- **风险 2：GitHub API 限流导致失败率高**
  - 应对：
    - 使用缓存与本地仓库镜像；
    - 适当引导用户错峰使用或上传压缩包作为后备方案（非 MVP）。

- **风险 3：运行步骤生成不准确，导致用户失败率高**
  - 应对：
    - 对典型技术栈编写规则或启发式逻辑，辅助模型；
    - 在教程中显式标注「可能因项目更新略有差异」。

- **风险 4：成本与性能**
  - 应对：
    - 对相同仓库的重复请求进行缓存；
    - 优先选择高性价比模型与嵌入方案；
    - 控制单次生成的长度。
